# AutoTrade Pro — Complete Project Documentation

> Paper Trading Only. This system uses virtual/simulated currency exclusively by default. No real money is ever involved unless every one of the following is true at once: `ZERODHA_PAPER_MODE=false`, `ZERODHA_ENABLED=true`, a valid Zerodha access token, and the request carries header `X-Confirm-Real-Order: yes`. ## Project Overview

AutoTrade Pro is a full-stack automated paper-trading platform for Indian markets (NSE/BSE). It continuously ingests price and news data, scores the tradeable universe with a multi-factor engine, reasons about the top candidates with an LLM, checks the result against a risk manager, and opens or manages simulated positions against a virtual wallet. A separately gated Futures and Options engine can trade index spreads, straddles, condors, and futures on top of the same infrastructure.

Highlights of what actually runs today:

- Master Intelligence Hub (`engine/intelligence_hub.py`) — builds one unified market context (macro, sector, news, earnings) and scores the NSE universe (turnover >= Rs 1 Cr/day, roughly 3,000 symbols) every 15 minutes. Two weight profiles exist (normal and swing), both dominated by technical score, with earnings/fundamental/options factors computed but currently zero-weighted.
- Three semi-independent paths can each open a live paper position: the Master Intelligence Cycle (inline scoring and execution, every ~15 minutes), the India Trade Loop (every 60 seconds, reads the same score table independently), and a News-First Discovery Engine (24/7 RSS polling and multi-tool LLM debate) that is built but, as of this writing, not wired to execution and not running as a service.
- A separate, older standalone Signal Engine (`engine/india_signal_generator.py`) still exists and is still live — it powers a legacy shortlist scanner, a couple of API routers, the older backtester, and a manual "run one agent cycle now" admin endpoint. It is not part of the automatic trading loop described above.
- LLM reasoning throughout the app — trade decisions, deep analysis, portfolio diagnostics, the AI chat assistant, earnings summaries, F&O commentary — is generated by a single provider: Mantle (AWS Bedrock), model `openai/gpt-oss-120b`. There is no Ollama, Groq, or Gemini fallback chain; those were removed.
- Futures and Options trading (index spreads, straddles, iron condors, futures, and a portfolio hedge) is fully built but gated off by default behind `ENABLE_FNO`, `ENABLE_OPTIONS`, `ENABLE_FUTURES`, `FNO_VOL_ENABLED`, and `FNO_HEDGE_ENABLED`.
- Capital sizing uses a conviction- and VIX-scaled formula (`capital_utilization_size`) rather than a fixed position count — see Risk Management below.
- Two distinct portfolios exist, clearly separated: the paper-trading Simulator (`/portfolio`) and My Portfolio (`/portfolio-tracker`), which unifies manually entered holdings, mutual funds, and Zerodha-synced holdings.

All data flows through a FastAPI backend with Celery workers on top of a local Postgres database; the React SPA talks to it over REST and WebSocket.

## Master Intelligence Hub

`engine/intelligence_hub.py` builds one `MasterContext` (macro bias, sector rotation, news score, earnings tone) and scores every symbol in the Hub universe.

Hub universe (`engine/hub_universe.py`): all NSE equities with 30-day average daily turnover >= Rs 1 Cr, ranked by `AVG(volume * close)`, capped at roughly 3,000 symbols, rebuilt daily. The threshold has been lowered progressively from Rs 20 Cr to Rs 5 Cr to Rs 1 Cr so that small-caps that move on real volume are never invisible to the scorer. It falls back to 1-hour candle aggregation if daily candles are thin. The breakout screener and momentum-discovery scanner inject qualifying movers into this universe live, independent of the daily rebuild.

Two weight profiles, both dynamically renormalized (the raw weights sum to 1.12, not 1.0 — the numbers below are the true effective, post-normalization weights):

Normal mode:

- Technical 58%
- Volume 13%
- News 11%
- Sector 9%
- Macro 9%
- Earnings, fundamental, and options factors are computed but weighted at 0% — not yet live in the score.

Swing mode (active during a swing-favorable regime):

- Technical 49%
- Sector 13%
- Volume 13%
- News 11%
- Macro 9%
- 12-month momentum 4%

A flat -20 penalty is applied to any symbol's score when the Nifty macro regime reads BEAR.

Signal thresholds on the resulting composite score: STRONG_BUY at 60 or higher (40 or higher in swing mode), BUY at 25 or higher, NEUTRAL between -25 and 25, SELL at -25 or lower, STRONG_SELL at -60 or lower.

Scores are persisted to the `MasterIntelligenceScore` table every cycle (~15 minutes) and are the live source both automatic decision paths read from — see the next section.

## Decision Architecture — How a Trade Actually Gets Placed

This is the part of the system that changed the most recently, and where the old documentation's "Decision Router" description was the most wrong.

### The Decision Router is not actually the routing layer

`engine/decision_router.py` exists and its docstring describes it as the single source of truth for paper/live routing. In the current code, its `route_decision()` function has zero callers anywhere outside its own file — it is not wired into any live path. Only `resolve_mode()` from the same file is used, and only by the `/api/v1/settings/mode` endpoints that back a UI toggle.

The actual paper/live gate is a plain boolean read directly by the execution code:

- `settings.AGENT_PAPER_MODE` — read in `engine/agent/execution.py` and `engine/agent/agent_loop.py`.
- A separately named `settings.PAPER_MODE` flag is also read in a couple of places in `agent_loop.py` and `engine/risk_manager.py`.
- A third, DB-backed `RuntimeConfig.paper_mode` value (table `RuntimeSettings`, key `paper_mode`) is what the `/settings/mode` toggle actually flips, and is what `resolve_mode()` reads.
- Real-money Zerodha order placement additionally checks its own `settings.ZERODHA_PAPER_MODE` flag before allowing any live order endpoint to run.

Nothing in the codebase guarantees these flags stay in sync with each other. Treat this as a known inconsistency rather than a single unified toggle — see Known Constraints below.

### Three paths can independently open a position

Path A — Master Intelligence Cycle (`tasks.run_master_intelligence_cycle`, Celery beat fires roughly every 15 minutes during market hours). In one cycle it: rebuilds the live snapshot cache, builds the hub universe, scores it, persists scores and daily history, runs a Tavily research gate on the top 15 BUY candidates, then evaluates its own top candidates inline using `StrategySelectorAgent`, `DecisionEngine.fuse()`, the LLM reasoning gate, `RiskManagerAgent.can_take_trade()`, and `AgentExecutionManager.execute()` — all in the same function call, with an overlap guard that skips a tick if the previous cycle is still marked "running." Capped at a handful of new candidates per cycle plus a daily new-entry cap.

Path B — India Trade Loop (`tasks.india_trade_loop`, every 60 seconds during NSE hours). This does not call the same code as Path A. It queries `MasterIntelligenceScore` directly (rows from the last 45 minutes — the same table Path A just wrote), then runs its own `validate_signal()` / `calculate_position_size()` from `engine/risk_manager.py` (a different module from `engine/agent/risk_manager.py`), its own LLM reasoning-gate call, and opens trades through `paper_trading.trade_simulator.open_paper_trade()`. Before looking for new entries each cycle it also: closes any SL/TP hits, runs LLM-driven dynamic SL/TP management on open positions (`engine/agent/dynamic_management.py`), checks drawdown circuit breakers (halting all new entries for the rest of the day if tripped), and checks a transient market-shock cooldown.

Path C — News-First Discovery Engine (`news_discovery_engine.py`, meant to run as the `news-engine.service` systemd unit). Polls free RSS every 15 seconds, keyword-filters headlines, extracts an NSE ticker with a cheap LLM call, then runs a multi-tool ReAct-style "debate" LLM (fundamentals, news, options, price action, market depth, sector, macro, and candle-prediction tools) that returns a TAKE or SKIP verdict. On TAKE it currently only logs the decision — it does not call any order/execution code, so it produces no real positions today. Outside market hours it queues candidates in `PreMarketNewsQueue` for the same script to drain at the next open. As of this writing the systemd unit is not installed, so this path is not running at all; verify with `systemctl status news-engine.service` and `pgrep -af news_discovery` before assuming otherwise.

### A fourth, manual, legacy path also exists

`engine/agent/agent_loop.py::run_agent_cycle()` is a full evaluation-and-execution cycle in its own right, but it is not on any Celery beat schedule. It is invoked directly (not via Celery) from `POST /api/v1/agent/run-cycle` (admin JWT required, since it mutates the paper book) for a manual "run one cycle now" trigger. Its symbol universe comes from a different source than Paths A and B: the `MarketShortlist` table (populated every 15 minutes by the separate legacy `tasks.market_scanner.run_market_scanner` task, which in turn is powered by the standalone `india_signal_generator.py` — see the next section), plus the user's watchlist, plus a hard-coded large-cap fallback. It drives the same four-strategy `StrategySelectorAgent` used by the AI Trading Agent (see below).

## Signal Engine and Technical Indicators

Two scoring systems coexist and are not fully unified:

1. `engine/intelligence_hub.py` — the universe-wide Hub score described above. This is what drives the two automatic live paths (A and B).
2. `engine/india_signal_generator.py` — an older, standalone signal engine with its own 15-input weighting, still live and still called from `api/india.py`, `api/signals.py`, the manual/legacy path above, and `engine/backtester.py`. Its weights: candlestick patterns 20%, technical indicators 35% (RSI, MACD, Bollinger Bands, EMA, ATR, Stochastic, Supertrend, VWAP, Ichimoku, ADX, EMA ribbon), FII/DII flow 20%, news 10%, sector rotation 10% (PCR for index symbols), India VIX 5%, plus a -30% damping modifier when an RBI Monetary Policy Committee meeting falls within 3 days. Public entry points: `generate_india_signal()` and `analyze_all_india_symbols()`.

Shared indicator math lives in `engine/indicators.py`, used by both systems and by Avishk chat: RSI, MACD, Bollinger Bands, EMA trend and EMA ribbon, Stochastic, Supertrend, VWAP, Ichimoku, ADX, a Fibonacci-retracement bonus, a momentum-breakout score, composite and swing-composite scoring functions, `score_to_signal()`, and ATR-based `suggest_stop_loss()` / `suggest_take_profit()`. The main entry point is `compute_indicators(df) -> IndicatorSignals`.

`engine/deep_analysis.py` powers `GET /api/v1/zerodha/deep-analysis/{symbol}` — a per-symbol, on-demand deep dive, distinct from the universe-wide Hub score. ## AI Trading Agent — Strategies, Regime, Backtesting

`engine/agent/selector.py::StrategySelectorAgent` runs four strategies that are actually wired into live execution: `PullbackTrendLong`, `MeanReversionShort`, `ExhaustionShort`, and `HubSignalStrategy` (evaluated last, as the widest net). The selector picks the highest-confidence candidate with a risk-reward ratio of at least 1.5 (Varsity module 9). Two additional strategies exist in code but are explicitly disabled based on backtest evidence recorded in code comments: `TREND_BREAKOUT_LONG` (mean R of -0.003 over 400-plus trades — no statistical edge) and `RANGE_REVERSAL_LONG` (only 2 trades, mean R of -0.336).

Morning regime classification (`engine/agent/morning_regime.py`) runs once per trading day via a single LLM call and sets the day into one of three modes: WAIT (no new entries — downtrend or high fear), SELECTIVE (only the highest win-rate strategy runs), or AGGRESSIVE (all strategies run). Separately, `engine/agent/market_regime.py` computes a continuous 5-state regime (STRONG_BULL, MODERATE_BULL, SIDEWAYS, WEAK_BEAR, STRONG_BEAR) from a composite of EMA stack, 20-day rate of change, EMA slope, market breadth, and India VIX; SIDEWAYS requires 82+ confidence, and WEAK_BEAR/STRONG_BEAR fully block new BUY entries.

Every strategy candidate, regardless of confidence, must still clear the risk manager's veto (see below) before it can become a trade — this is the "unconditional risk-manager veto" the code comments describe.

Two backtesters exist side by side:

- `engine/agent/backtester.py` (`AgentBacktester`) — vectorized event-bar replay using the same `StrategySelectorAgent` and `MarketAnalyzerAgent` as live trading, with a realistic Indian cost model (brokerage `min(Rs 20, 0.03%)`, STT 0.1%, exchange charge 0.00345%, SEBI fee 0.0001%, stamp duty 0.015% on buys only, 18% GST on fees).
- `engine/backtester.py` — an older backtester that replays signals from `india_signal_generator.py` with ATR-based stops.

## Risk Management

`engine/agent/risk_manager.py::RiskManagerAgent.can_take_trade()` runs an ordered gate sequence — a candidate must clear all of these to become a trade:

1. Circuit breakers — daily drawdown stop at 3%, weekly at 5%, monthly at 10%.
2. Behavioral locks (Varsity module 12) — skipped entirely in paper mode by design, since paper mode is meant to demonstrate the full scan output. In live mode: a consecutive-loss lockout after 2 losses, and a cap of 20 new entries per day.
3. Position sizing via `capital_utilization_size()` — see below; a zero-quantity result rejects the trade.
4. Oversize-trade guard — single-trade risk must stay within `AGENT_MAX_RISK_PER_TRADE` (1% of equity).
5. Portfolio open-risk cap — total open risk across all positions must stay within `AGENT_MAX_OPEN_RISK` (15% of equity).
6. Cash buffer — post-trade cash must remain at least 20% of equity.
7. Duplicate-position guard, matching on NSE/BSE-suffix-normalized symbol.
8. Correlation-cluster guard (Varsity module 16) — blocks the trade if its correlation with any already-open symbol exceeds 0.70.
9. Sector exposure — two independent caps: notional exposure per sector capped at 20% of equity, and position count per sector capped at 2.
10. Confidence gate — the candidate's confidence must be at least `AGENT_CONFIDENCE_THRESHOLD` (30).
11. Minimum reward-to-risk ratio of 1.5, re-checked here.

Two blunt safety ceilings sit outside the gate function itself, in configuration: `MAX_OPEN_POSITIONS = 20` (explicitly commented as a bug-guard safety ceiling, not the primary limiter) and `MAX_DAILY_LOSS = 5%`, which halts all trading for the day if breached.

Capital sizing (`capital_utilization_size()`, used by Path A and by the manual/legacy path): position weight scales from 2% to 5% of equity as conviction rises toward the configured high-conviction threshold, damped by India VIX (a multiplier of 1.0 at VIX 22 down to 0.5 at VIX 30), bounded by the 5% hard per-position cap, the 20% cash-buffer room, and the 1% max-risk-at-stop guard. Path B uses its own, separately implemented `validate_signal()` / `calculate_position_size()` in `engine/risk_manager.py` rather than this function.

The paper-trading wallet's starting balance is DB-configurable via `RuntimeConfig.paper_trading_balance` rather than a hardcoded constant — Rs 20L is the current default, illustrated directly in a code comment as "Rs 1L on Rs 20L" for the 5% cap example.

## Paper Trading Simulation

`paper_trading/trade_simulator.py` and `paper_trading/virtual_wallet.py`:

- Every fill applies the same realistic Indian cost model used by the backtester (brokerage, STT, exchange charge, SEBI fee, stamp duty, GST).
- Slippage is modeled as a random 0.01% to 0.03% of entry price, applied adversely on entry.
- Cash is charged at the post-slippage fill price, not the raw signal price — a documented fix for a bug where the pre-slippage price understated cost basis and inflated paper equity.
- Equity stop-loss/take-profit checks run via the `fast_sl_check` Celery task every 5 seconds, restricted to NSE trading hours, exit-only (it never opens new positions), and reads live price directly from the Kite REST API rather than the in-process price cache, because Celery worker processes do not share the FastAPI process's in-memory cache.
- `VirtualWallet.check_drawdown_breakers()` trips a sticky `trading_halted` flag on a mark-to-market daily loss beyond the configured maximum, blocking all new entries (but not exits) until cleared.

## News and Sentiment

`crawler/news_crawler.py` runs FinBERT (`ProsusAI/finbert`) as the core sentiment model, with a keyword-based guard layer applied before FinBERT sees a headline — Indian broker-recommendation language ("BUY", "SELL", "ACCUMULATE") and wait-and-see/uncertainty phrasing both get pre-classified, because FinBERT tends to mis-score both patterns on its own. If `torch`/`transformers` are unavailable, a pure keyword fallback takes over instead. News sources are tried in a tiered order: free RSS feeds first (India-first, no API key needed), then yfinance news (already ticker-tagged), then NewsData, NewsAPI, and Finnhub. The resulting sentiment score is persisted per headline on `NewsItem.score` — this is what the Hub's news factor reads.

`engine/tavily_enricher.py` is a separate enrichment layer, not the primary sentiment source: it fills news gaps for small-caps with zero RSS/DB coverage (once per Hub cycle, only for symbols with a zero news score) and generates deep-research summaries for the top 5 Telegram shortlist alerts. It is budget-capped at roughly 1,000 free credits per month, runs fully async, and silently swallows failures so it never blocks the main pipeline.

`engine/narrative_engine.py` is a separate top-down "Eagle Eyes" sector engine: it scrapes Telegram and RSS every 5 to 10 minutes, uses Mantle (`gpt-oss-120b`) to extract currently hot sectors and keywords, and writes an in-memory boost cache that adds up to +20 to the score of any stock in a hot sector. This is the same narrative boost referenced in the F&O signal engine's news factor.

## LLM Integration

`utils/llm.py` — the sole LLM provider across the entire application is Mantle (AWS Bedrock), model `openai/gpt-oss-120b`, invoked through `call_llm_chat()`. There is no Ollama, Groq, or Gemini fallback chain; legacy fallback parameters are still accepted by the function signature for backward compatibility but are ignored — every call goes to gpt-oss now.

Reasoning traces are captured and persisted for every decision-relevant LLM call, via a context-var-based mechanism (`log_llm_reasoning`) writing to the `LLMReasoningLog` and `reasoning_verdicts` tables. The full trace for agent decisions is surfaced on the `/agent-log` page. This spans the trade decision gate, Deep Analysis, Portfolio Doctor's narrative, Avishk chat, the Earnings Call Analyzer, F&O AI desk analysis, and the narrative engine's sector extraction.

## Futures and Options Engine

Fully built, gated off by default behind `ENABLE_FNO`, `ENABLE_OPTIONS`, `ENABLE_FUTURES`, `FNO_VOL_ENABLED`, and `FNO_HEDGE_ENABLED` (all default `False`). Covers NIFTY, BANKNIFTY, and FINNIFTY. See the `/fno-pipeline` page in the app for a full interactive breakdown; summarized here:

- Greeks are computed with a native scipy Black-Scholes-Merton implementation (`engine/fno/options_pricing.py`) — delta, gamma, vega, theta, and rho, with implied volatility solved via Brent's method.
- Contract resolution tries the live Kite instrument master first, then falls back to a snapshot-derived synthetic contract for paper mode, with a minimum 2 days to expiry to avoid expiry-day gamma risk.
- Two independent evaluation passes run per cycle. The directional pass (`evaluate_index_options()`) opens a Bull Call Spread or Bear Put Spread from the same 7-factor composite index signal used for the Signals panel (price trend/momentum, PCR contrarian, FII/DII flow, market breadth, max-pain gravity, news sentiment, and a VIX-based conviction damper), gated by a 55% confidence threshold and blocked against opening against the current market regime. It never checks IV-Rank. The volatility pass (`evaluate_volatility()`, gated separately by `FNO_VOL_ENABLED`) opens a Long Straddle when IV-Rank is below 30 or an Iron Condor when IV-Rank is above 70, and never checks the directional signal.
- Exit rules differ by strategy: spreads take profit at 50% of max profit and stop at 80% of max loss; straddles stop at -50% and take profit at +100% per leg; Iron Condor legs are currently opened with no stop-loss or take-profit at all, so they ride to expiry.
- A daily expiry sweep (weekdays, 3:45 PM IST) settles options at intrinsic value and futures at spot, returning margin to the wallet.
- A portfolio hedge (`evaluate_portfolio_hedge()`, gated by `FNO_HEDGE_ENABLED`) buys a protective NIFTY put when the open equity book exceeds 10% of equity and the index signal turns bearish.

## Portfolio Doctor

`engine/portfolio_doctor.py` runs exactly seven diagnostic modules: concentration, risk quality, diversification, tax efficiency, performance, sector timing, and position sizing. Scoring starts at 100 and applies per-finding deltas (CRITICAL -25, WARNING -10, INFO -3, GOOD +2), clamped to the 0-100 range, then maps to a letter grade: A at 85 or above, B at 70, C at 55, D at 40, otherwise F. The narrative explanation is generated through Mantle.

---

## Earnings Call Analyzer

`engine/earnings_summarizer.py` fetches BSE/NSE filed transcripts for any NSE-listed company via dynamic scrip resolution, extracts text from the PDF, and generates a structured AI summary through Mantle (`gpt-oss-120b`), including a management-tone rating and the reasoning behind it. A rule-based fallback summary is produced if the AI call fails.

## Zerodha KiteConnect Integration

`api/zerodha.py` exposes 60 endpoints under `/api/v1/zerodha` — OAuth login and callback, holdings, positions, orders, trades, P&L, order placement, live prices, market depth, token status, watchlist analysis, and margins. Every state-changing live-order endpoint checks `settings.ZERODHA_PAPER_MODE` and rejects the call with a 403 if paper mode is still on; real order placement additionally requires the `X-Confirm-Real-Order: yes` header.

## Unified Market Data Layer

`get_price()` in `crawler/live_prices.py` is the single source of truth for a live quote: it tries the Zerodha KiteTicker WebSocket cache first (accepted up to 30 seconds old), then the 15-second-refreshed price cache backed by yfinance (also up to 30 seconds old), then gives up. It always returns a price with its source and age in seconds, or `None`. A separate `yfinance_ltp_batch()` function is a process-independent batch fetcher added because Celery worker processes don't share the FastAPI process's in-memory price cache or WebSocket connection — it backstops the fast stop-loss checker and the shock guard.

## Celery Background Tasks

Selected schedule, all times converted to IST (celery beat times are stored in UTC):

- Continuous: price scan every 30 seconds; live-price cache refresh every 15 seconds; fast stop-loss check every 5 seconds (market hours only).
- Every 5 minutes: narrative/macro intelligence refresh; breakout screener (NSE-open gated, offset 60 seconds after each mark).
- Every 15 minutes: options chain refresh; the legacy market scanner that populates `MarketShortlist`; the Master Intelligence Cycle (Path A — scoring plus inline execution, offset 45 seconds after each bar close).
- Every 30 minutes: momentum discovery scan — runs around the clock, not gated to market hours.
- Every 60 seconds: the India Trade Loop (Path B), during NSE hours.
- Daily 9:00 AM IST: hub universe rebuild.
- Daily around 3:25 PM IST: agent end-of-day reconcile.
- Weekdays 3:45 PM IST: F&O expiry sweep.
- Various other daily/weekly jobs: FII/DII fetch, corporate-action checks, weekend reflection, weekly ML model retraining, weekly portfolio rebalance report, and instrument/token refresh jobs for Zerodha.

The News-First Discovery Engine (Path C) is not on this schedule at all — it is meant to run as its own always-on systemd service, independent of Celery beat.

## Infrastructure

Only Postgres actually runs in Docker (`autotrade_postgres`, Postgres 17-alpine, port 5432). `docker-compose.yml` also defines `app`, `celery_worker`, `celery_beat`, and an Adminer database browser on port 8080, but that is not how the backend is actually deployed day to day. The real deployment is three `systemd --user` services: `autotrade-uvicorn.service` (API, no auto-reload), `autotrade-celery-worker.service`, and `autotrade-celery-beat.service` (both of the latter auto-restart on any `.py` file change via `watchmedo`), plus a `autotrade-zerodha-refresh.timer` firing daily at 8:00 AM IST. Redis runs as a bare host process, not in Docker.

## Development Setup

The backend virtual environment lives at `autotrade-backend/.venv`, Python 3.11. The host's bare `python3` is a newer, incompatible version (3.14) — running backend code with it breaks `pydantic_core` due to an ABI mismatch, so always invoke `.venv/bin/python` directly. `start.sh` checks for the systemd services described above first and falls back to a manual `pkill`/relaunch if they aren't present; if the venv itself is missing it prints a remediation hint (`python3.11 -m venv .venv && .venv/bin/pip install -r requirements.txt`).

## Known Constraints and Design Decisions

- Paper trading is the default and the safety rail — see the warning at the top of this document. Because there are three independently-read "paper mode" flags (`AGENT_PAPER_MODE`, `PAPER_MODE`, and DB-backed `RuntimeConfig.paper_mode`) plus a fourth Zerodha-specific one (`ZERODHA_PAPER_MODE`), don't assume flipping one toggle in the UI changes trading behavior everywhere — verify which flag actually gates the path you care about.
- `engine/decision_router.py` reads as the architectural "front door" for trade routing but isn't actually called by the live paths — treat its docstring as aspirational, not current behavior.
- Two independent signal-scoring systems exist side by side (the Hub score and the older standalone `india_signal_generator.py`). They are not reconciled with each other and can, in principle, disagree about the same symbol.
- The News-First Discovery Engine is built end-to-end but, as shipped, does not place real (even paper) trades on a TAKE verdict, and its systemd service is not installed by default — verify its live status with `systemctl status news-engine.service` before relying on it.
- Futures and Options trading is fully implemented but gated off by five separate flags, all defaulting to `False` — nothing F&O-related executes until an operator deliberately enables it.
- `MAX_OPEN_POSITIONS = 20` and `MAX_DAILY_LOSS = 5%` are described in their own code comments as blunt safety ceilings, not the primary risk controls — the real position sizing and risk gating happens in the 11-step `RiskManagerAgent` sequence documented above.
